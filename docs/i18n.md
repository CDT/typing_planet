# Internationalization

Locale strategy, key conventions, and font fallback. UI strings are translated; lesson typing content is **not** translated (it stays English in v1.0 since that's the touch-typing skill being taught).

## Supported Locales (v1.0)

| Locale  | Name        | Status   |
| ------- | ----------- | -------- |
| `zh-CN` | 简体中文    | Default  |
| `en-US` | English     | Full     |

Default is `zh-CN`. Locale is persisted in settings and survives reloads. First-launch default uses `navigator.language` matched against the supported list; falls back to `zh-CN`.

`html[lang]` is updated reactively when the user toggles locale.

## Library

`react-i18next` with these settings:

```ts
i18n.use(initReactI18next).init({
  fallbackLng: "zh-CN",
  lng: settingsStore.getState().locale,
  resources: {
    "zh-CN": { translation: zhCN },
    "en-US": { translation: enUS }
  },
  interpolation: { escapeValue: false },
  returnNull: false
});
```

ICU plural and numeric formatting is enabled via `i18next-icu`.

## File Layout

```
src/i18n/
├── index.ts
├── zh-CN.json
└── en-US.json
```

Both files are bundled at build time (no lazy loading in v1.0; combined size ≤ 20 KB gzipped — too small to be worth splitting).

## Key Conventions

- Flat dot-notation namespaces: `<page>.<section>.<element>`. E.g. `map.header.greeting`, `practice.hud.wpm`, `settings.data.reset.confirm`.
- Pluralization uses the i18next syntax: `practice.errors_one` / `practice.errors_other`.
- Interpolation: `{{wpm}}` for variables, never positional `%s`.
- No raw strings in JSX. ESLint rule `react/jsx-no-literals` is enabled with an allowlist for symbol-only strings (`"·"`, `"—"`).

### Example

```json
// zh-CN.json
{
  "map": {
    "header": {
      "greeting": "你好",
      "streak_one":  "{{count}} 天连续练习",
      "streak_other":"{{count}} 天连续练习"
    }
  },
  "practice": {
    "hud": { "wpm": "字/分钟", "accuracy": "准确率" },
    "errors_one":   "{{count}} 个错误",
    "errors_other": "{{count}} 个错误"
  },
  "results": {
    "stars_one":   "{{count}} 颗星",
    "stars_other": "{{count}} 颗星"
  }
}
```

```json
// en-US.json
{
  "map": {
    "header": {
      "greeting": "Hi there",
      "streak_one":  "{{count}}-day streak",
      "streak_other":"{{count}}-day streak"
    }
  },
  "practice": {
    "hud": { "wpm": "WPM", "accuracy": "Accuracy" },
    "errors_one":   "{{count}} error",
    "errors_other": "{{count}} errors"
  },
  "results": {
    "stars_one":   "{{count}} star",
    "stars_other": "{{count}} stars"
  }
}
```

## Authoring Rules

- Both locales must have the same set of keys. CI runs `pnpm i18n:check` (compares key trees) and fails on mismatch.
- Source-of-truth language is **Chinese**. New strings are added in `zh-CN.json` first; `en-US.json` follows.
- Lesson titles live in lesson JSON (`title.zh-CN`, `title.en-US`) — not in the i18n bundles.
- Avoid embedding HTML in translation strings. If markup is needed, use `<Trans i18nKey="..."/>` with components.

## Numbers, Dates, Plurals

- Use `Intl.NumberFormat` and `Intl.DateTimeFormat` with the current locale — no date-fns/locale bundles.
- Streak day count and lesson "best WPM" use locale-appropriate digit grouping.
- Streak boundaries use the **user's local timezone** (`Intl.DateTimeFormat().resolvedOptions().timeZone`), not UTC.

## Typography & Fonts

- Latin text uses bundled Inter + JetBrains Mono.
- CJK characters fall back to: `"PingFang SC"`, `"Microsoft YaHei"`, `"Source Han Sans SC"`, system-ui.
- We do **not** bundle CJK webfonts — they're too large for the [performance budget](./performance-budget.md).
- The typing surface uses a monospace stack: `"JetBrains Mono", "Sarasa Mono SC", monospace`. Since lesson text is English in v1.0, the latin monospace is what's actually rendered.

## RTL

Out of scope for v1.0 (no RTL languages supported). The layout uses CSS logical properties (`padding-inline-*`, `margin-inline-*`, `text-align: start`) so adding Arabic or Hebrew later is mostly a translation effort.

## Lesson Content & Locale

Lesson content (`content[]` lines) is **not** localized — it's the English text the user types. Lesson **titles and descriptions** are localized via `title.<locale>` in the lesson JSON. See [content-spec.md](./content-spec.md) §lesson-json-schema.

When CN typing content lands in a later version (see [roadmap.md](./roadmap.md)), the lesson schema will gain a `language` field; the engine and the UI will branch on it.

## Translation Workflow

1. Developer adds a new key to `zh-CN.json` with the Chinese source string.
2. The same key with an English value is added to `en-US.json`.
3. `pnpm i18n:check` confirms parity.
4. PR review includes a spot-check of the English translation; if unsure, leave a `// TODO(i18n)` comment and tag the reviewer.

No external TMS in v1.0. If contributors grow, migrate to Crowdin/Lokalise.
